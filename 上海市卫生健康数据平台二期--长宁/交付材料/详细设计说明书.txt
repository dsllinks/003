1.2项目背景
　　　“十四五”时期是全民健康信息化建设创新引领卫生健康事业高质量发展的重要机遇期，也是以数字化、网络化、智能化转型推动卫生健康工作实现质量变革、效率变革、动力变革的关键窗口期。2021年3月，《中华人民共和国国民经济和社会发展第十四个五年规划和2035年远景目标纲要》（以下简称“纲要”）发布，纲要中指出：提高数字政府建设水平。将数字技术广泛应用于政府管理服务，推动政府治理流程再造和模式优化，不断提高决策科学性和服务效率。在智慧医疗领域，完善电子健康档案和病历、电子处方等数据库，加快医疗卫生机构数据共享。推广远程医疗，推进医学影像辅助判读、临床辅助诊断等应用。运用大数据提升对医疗机构和医疗行为的监管能力。
　　　2023年4月，上海市城市数字化转型工作领导小组办公室关于印发《2023年度上海市“便捷就医服务”数字化转型3.0工作方案》（沪数字化办[2023]2号）的通知，基于卫生健康行业原有成果，充分运用云计算、大数据、人工智能等数字化技术，赋能智慧医疗创新应用场景，为提升诊疗水平、提供医疗服务提供数字化支撑，对接医疗数字化转型中的共性需求，筑牢医疗服务数字化转型底层基石，夯实卫生健康行业产业智慧化发展基础，实现“1+1+1+1”的立体式行业数字化基座。
　　　2021年10月，《长宁区全面推进城市数字化转型 “十四五”规划》（长科委〔2021〕22号），数字健康方面，深化拓展人口健康信息化工程，1家社区卫生服务中心达到EHR测评6级标准，7家通过了5级测评，2家通过了4级测评。建设分级诊疗服务管理平台，为家庭医生、业务专家和管理人员提供统一的信息平台，提供居民签约、统一服务入口。重点围绕三大系统建设，构建支撑系统运行服务的移动、互联网、区域医疗三网融合的网络环境，直接对接云HIS，实现移动端挂号、实时查询临检中心报告、支持家床用药的处方外送，延伸处方等功能服务。升级改造区卫生应急指挥中心，建立基于信息化的医防联动应急处置平台;推动智能助医项目建设，打造智能助医应用服务体系，赋能社区卫生服务机构与基层医生，进一步提升社区卫生健康综合服务能力。
　　　综上，在“十四五”时期，全民健康信息化建设将迎来重要机遇期。2021年3月，《中华人民共和国国民经济和社会发展第十四个五年规划和2035年远景目标纲要》提出了提高数字政府建设水平的目标，并在智慧医疗领域做出了明确规划。2022年11月，《“十四五”全民健康信息化规划》发布，确立了全民健康信息化发展目标，包括初步建设全民健康信息平台支撑体系、加速推进全民健康信息化基础设施建设等。此外，各级政府也相继出台了一系列支持数字化转型的政策文件和实施方案。在地方层面，上海市和长宁区都制定了城市数字化转型规划，并在数字健康方面做出了积极探索和努力。例如，上海市通过建设智慧医疗应急体系和推动数字健康城区建设等举措，促进了数字健康服务的发展。长宁区则深化了人口健康信息化工程，建设了分级诊疗服务管理平台，提升了社区卫生健康综合服务能力。未来的医疗健康数字化转型建设应以更高的标准、更严格的要求，推动卫生健康行业迈向数字化转型，为人民群众提供更加便捷、高效、优质的医疗健康服务。

2建设目标
　　　长宁数智基座，作为长宁区医疗数字化转型的核心战略和支撑平台，奉行“统一规划，分步实施”的策略，共计分为三期完成建设。旨在通过整合资源、优化服务，提升医疗卫生领域的数字化水平和服务质量，实现全民健康信息化管理的目标。到2025年，该基座已初步建设形成了统一权威、互联互通的支撑保障体系，为公立医疗卫生机构提供了全面的技术支持与信息共享平台，确保医疗服务的全覆盖和高效运行。
　　　通过数字化技术的应用，医疗卫生服务得以更广泛地渗透到社会各个角落，业务各环节实现了更紧密的连接与协同，智能化的管理与应用提升了工作效率与服务质量，数据的集约共享使得医疗资源得到更加合理的配置，安全可控的体系保障了医疗信息的隐私与完整性。
　　　长宁区全民健康信息统筹管理能力显著增强，医疗卫生机构间的信息互通共享取得了标志性进展。未来通过数智基座的支持，数字健康服务已成为医疗卫生服务的重要组成部分，为每个居民建立了一份动态管理的个性化电子健康档案，实现了医疗信息的全面管理和共享。同时，卫生健康行业机构的数字化、资源网络化、服务智能化和监管一体化等方面基本形成，为长宁区建设全面的全民健康信息服务体系奠定了坚实基础。一期建设总体目标如下：
　　　一期夯实卫生健康数智基座基础，为医疗服务人员减负提效，提升医疗卫生监管能级。针对数据烟囱、系统流程不连续、医疗资源不均衡、监管滞后等突出问题，一期将重点完成数智基座的建设，统一全区数据采集标准规范，实现所有机构产生的医疗卫生数据的汇聚、清洗与转换，实现跨机构、跨系统数据共享，提升临床诊疗效率。同时以基座数据为基础，一是进一步增加转诊疾病种类和丰富转诊内容，让居民就近享受到同仁、天山等优质资源；二是面向管理人员，多维度展示全区医疗运行状况，为本区卫生健康政策研究、制定相关政策提供决策依据。

4.2.3.1.业务场景分析
　　系统模板功能为管理员提供了便捷的门户创建方式。管理员在获取平台账号和密码并成功登录统一门户管理后台后，进入系统模板界面，能够看到系统按照统一设计规范内置的丰富门户模板。这些模板以卡片列表形式展示，管理员可以通过点击 “预览” 按钮查看每个模板的详细样式，了解其布局和内容。当管理员需要创建新的门户时，可基于已有的系统模板进行操作，点击 “复制” 按钮，新门户将继承所选系统模板的样式布局，随后管理员可以在门户设计页面进行样式调整，如修改部分内容、调整页面元素等，最后保存设置，形成适用于当前机构的独特门户样式。由于系统模板由平台提供且不允许编辑，这保证了模板的规范性和一致性，同时也为管理员在创建门户时提供了标准化的基础样式，节省了设计时间和精力，提升了门户创建的效率和质量。

4.2.4.我的门户模块详细设计
4.2.4.1.业务场景分析
　　　“我的门户” 功能为交付人员提供了灵活的门户配置能力。交付人员登录统一门户管理后台后，可进入 “我的门户” 界面，查看基于系统模板或已有模板创建的门户列表。这些门户以卡片形式展示，支持按名称模糊搜索和创建时间倒序排列。交付人员能通过 “预览” 按钮查看门户样式，对 “待使用” 状态的门户可点击 “编辑” 跳转至设计页面修改内容。若需复用已有门户样式，可点击 “复制” 继承布局并调整后保存为新样式。当门户样式配置完成，交付人员通过 “使用” 操作，在弹出的机构树弹窗中选择当前机构及子级机构范围，使所选门户在对应机构生效为 “使用中” 状态，且每个机构仅允许一个 “使用中” 门户，确保前台用户登录时看到统一规范的界面。此外，上级机构门户禁止编辑，保障了整体架构的稳定性与规范性，助力交付人员高效管理多机构的门户定制需求。

4.2.1.3.6.1测试条件
4.2.1.3.6.1.1功能测试
测试计划：
a)  数据列表页面展示正常，展示数据信息

4.2.1.1.20.1.2性能测试
测试条件：系统在1000个并发用户下的响应时间和资源利用率。
测试步骤：
使用压力测试工具模拟1000个用户同时访问系统。
记录系统的平均响应时间、吞吐量和服务器资源利用率。
分析测试结果，评估系统在高负载情况下的性能表现。
4.2.1.1.20.2限制条件
4.2.1.1.20.2.1技术限制
限制条件：系统支持浏览器是IE9以上版本，chrome浏览器，火狐浏览器，360浏览器(极速模式) 。
影响范围：用户使用其他浏览器可能无法正常访问系统。
解决方案：向用户提供浏览器升级提示，并在系统界面中标明最佳浏览器推荐。
4.2.1.1.20.2.2数据库限制
限制条件：数据库连接池最大连接数为1000。
影响范围：系统在高并发情况下可能出现数据库连接超时。
解决方案：优化数据库查询语句，增加数据库连接池的最大连接数。

4.2.4.5.数据质量异常监测详细设计
4.2.4.5.1.业务场景分析
　　　数据质量异常监测功能的业务场景主要围绕保障数据质量展开。质量管理员和数据开发者需要确保用于分析和决策的数据准确可靠。在日常数据处理流程中，质量管理员根据业务需求开发如指标校验类的质控规则，将其组装成质控方案并创建任务。当任务执行时，系统依据这些规则对大量的传染病监测数据进行检测。数据开发者则在任务运行后查看质控结果，了解数据的整体质量状况。质量管理员进一步通过多维度过滤查看详细数据，从机构整体得分到具体数据表的字段问题详情，层层钻取，精准定位异常数据。发现异常后进行标记，为后续的数据处理和质量提升提供明确指引，确保数据能有效支持传染病监测预警工作。

4.2.4.8.数据质量自动化详细设计
4.2.4.8.1.业务场景分析
　　　数据质量自动化功能的业务场景主要体现在保障数据的一致性和准确性上。在数据采集完成后，质量管理员需要确保采集到的数据符合质量标准。当面对跨源数据校验时，可能会出现数据不一致的情况，例如记录缺失或数据量不匹配等问题。此时，质量管理员通过配置自动补偿策略，为每张质控表设置对应的补偿任务并开启自动补偿功能。当系统检测到跨源校验中【记录缺失检查】规则的核对结果为 “错误”，且质控表数据量小于参照表数据量时，系统会自动按照当次质控数据的时间范围重新补采数据。补采完成后，质量管理员再刷新查看数据条数比对结果，从而保证数据的一致性。若未提前配置自动补偿策略，也可手动选择补偿任务来快速补偿数据，以满足传染病监测预警对数据质量的严格要求。

4.4.应用中心
4.4.1医技预约详细设计 
4.4.1.1 用户管理服务 
4.4.1.1.1 功能概述 
　　　负责用户的注册、登录、信息管理、权限控制等功能，包括医护人员、系统管理员二类用户。 
4.4.1.1.2 类设计 
1.User 类：封装用户基本信息，包括用户 ID、姓名、性别、年龄、联系方式、身份证号、用户类型（患者 / 医护人员 / 管理员）等属性，以及对应的 getter 和 setter 方法。 
2.UserService 接口：定义用户管理相关的业务方法，如register(User user)（用户注册）、User login(String username, String password)（用户登录）、void updateUser(User user)（更新用户信息）等。 
3.UserServiceImpl 类：实现 UserService 接口，调用 UserMapper 进行数据库操作，完成用户注册、登录验证、信息更新等业务逻辑。 
4.UserMapper 接口：继承 MyBatis-Plus 的 BaseMapper 接口，定义用户相关的数据库操作 SQL 语句，如根据用户名查询用户信息、插入新用户等。 
4.4.1.1.3 业务流程 
1.用户登记：用户提交注册信息，UserServiceImpl 接收信息，验证信息合法性（如用户名是否重复、密码格式是否正确），通过后调用 UserMapper 将用户信息插入数据库，注册成功返回用户 ID。 
2.用户登录：用户输入用户名和密码，UserServiceImpl 调用 UserMapper 查询数据库验证用户信息，验证通过后生成 Token（基于 JWT）返回给用户，后续用户请求携带 Token 进行身份验证。 


4.4.1.2 预约服务 
4.4.1.2.1 功能概述 
　　　为医生提供医技检查预约功能，包括检查项目查询、预约时段选择、预约订单生成、预约取消与改约等。 
4.4.1.2.2 类设计 
1.Appointment 类：表示预约订单，包含预约 ID、患者 ID、检查项目 ID、预约时间、预约状态（待检查 / 已完成 / 已取消）、预约医疗机构 ID 等属性，以及相关的查询方法。 
2.AppointmentService 接口：定义预约相关业务方法，如获取可预约时段）、（创建预约订单）、取消预约）等。 
3.AppointmentServiceImpl 类：实现 AppointmentService 接口，调用 AppointmentMapper 和其他相关服务（如查询服务、用户管理服务）完成预约业务逻辑。 
4.AppointmentMapper 接口：继承 BaseMapper 接口，定义预约订单相关的数据库操作 SQL，如插入预约订单、更新预约状态等。 
4.4.1.2.3 业务流程 
1.预约时段查询：患者选择检查项目后，AppointmentServiceImpl 调用调度服务获取该项目在各医疗机构的可预约时段，结合自身业务规则（如患者优先级、距离限制等）筛选出符合条件的时段返回给患者。 
2.预约创建：患者选择预约时段提交预约请求，AppointmentServiceImpl 验证患者身份和预约条件（如是否超过预约次数限制），通过后生成预约订单，调用 AppointmentMapper 插入数据库，并通知调度服务生成调度任务。 
3.预约取消与改约：患者发起取消或改约请求，AppointmentServiceImpl 判断预约状态是否允许操作，若允许则更新预约状态，调用调度服务释放或重新分配资源。 
4.4.1.3 调度服务 
4.4.1.3.1 功能概述 
根据预约订单和调度算法，为预约订单分配检查时间和设备，处理资源冲突，实现智能调度。 
4.4.1.3.2 类设计 
1.ScheduleTask 类：表示调度任务，包含任务 ID、预约 ID、检查时间、检查设备 ID、医技科室 ID 等属性，以及相关的 getter 和 setter 方法。 
2.ScheduleService 接口：定义调度相关业务方法，如 （为预约订单生成调度任务）、 处理调度冲突）等。 
3.ScheduleServiceImpl 类：实现 ScheduleService 接口，调用 ScheduleMapper 和其他相关服务（如预约服务、医技科室服务），运用调度算法完成调度业务逻辑。 
4.ScheduleMapper 接口：继承 BaseMapper 接口，定义调度任务相关的数据库操作 SQL，如插入调度任务、更新任务状态等。 
4.4.1.4.2 类设计 
1.DataShareService 接口：定义数据共享相关业务方法，如 （推送检查结果）、 查询共享数据）等。 
2.DataShareServiceImpl 类：实现 DataShareService 接口，通过调用数据库操作和消息队列，实现数据的推送和查询功能。 
4.4.1.3.3 业务流程 
1.调度任务生成：接收到预约服务的调度请求后，ScheduleServiceImpl 根据调度算法（如基于优先级的最早可用时间算法），查询医技科室的设备使用情况和工作负荷，为预约订单分配检查时间和设备，生成调度任务并调用 ScheduleMapper 插入数据库。 
2.冲突处理：定时扫描调度任务，若发现资源冲突（如同一设备同一时间被多个任务占用），ScheduleServiceImpl 根据冲突解决策略（如调整优先级低的任务时间），调用 ScheduleMapper 更新相关任务，必要时通知预约服务调整预约时间并告知患者。 
4.4.1.4 数据共享服务 
4.4.1.4.1 功能概述 
实现患者预约信息、检查结果等数据在区域内各医疗机构间的共享，提供数据推送和查询功能。 
4.4.1.4.3 业务流程 
1.检查结果推送：医技科室上传检查结果后，DataShareServiceImpl 接收到结果数据，根据患者信息和相关医生信息，通过消息队列将结果推送给对应患者和医生。 
2.数据查询：医护人员或患者发起数据查询请求，DataShareServiceImpl 根据查询条件，从数据库中查询相关共享数据并返回。 
4.4.1.5 统计分析服务 
4.4.1.5.1 功能概述 
对预约数据进行统计分析，生成各类报表，如每日预约量统计、各医疗机构预约占比统计、检查项目热门程度统计等，为管理决策提供依据。 
4.4.1.5.2 类设计 
1.StatisticsService 接口：定义统计分析相关业务方法，如 获取每日预约量统计）、 获取各医疗机构预约占比统计）等。 
2.StatisticsServiceImpl 类：实现 StatisticsService 接口，通过复杂的 SQL 查询和数据处理，生成统计报表数据。 
4.4.1.5.3 业务流程 
　　　系统管理人员或相关人员发起统计请求,StatisticsServiceImpl 根据请求类型，执行相应的 SQL 查询语句从数据库中获取数据，进行数据处理和计算，生成统计报表数据返回给请求方。
